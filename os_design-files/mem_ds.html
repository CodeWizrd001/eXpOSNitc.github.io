File Table



The File Table stores the information about all the files that are open while the OS is running. It consists of MAX_OPENFILE_NUM (= 32 in the present eXpOS version) entries. Thus, there can be at most MAX_OPENFILE_NUM = 32 open files in the system at any time.

The Open system call creates an entry in the File table when a process opens a file that is not opened by any process in execution. If the file is opened again by some other process (or the same process), the Open system call updates the same file table entry. The first field (Inode Index) of the File table entry for a file is set to the index of the Inode Table entry for the file. The second field (File Open Count) stores the number of open instances of the file and is incremented/decremented when a process opens/closes the file.

If the file open count reaches zero (that is if the last open instance has been closed) then, all fields of the corresponding file table entry is invalidated by setting to -1.

Each entry is of size 8 words of which the last 4 are left unused.

Every entry of the File Table has the following format :

INODE INDEX FILE OPEN COUNT ULOCK KLOCK Unused
  • INODE INDEX (1 word) - specifies the index of the entry for the file in the Inode table.
  • FILE OPEN COUNT (1 word) - specifies the number of open instances of the file
  • ULOCK (1 word)- If the file was locked by a process using FLock system call, this field specifies the PID of the process, otherwise, it is set to -1.
  • KLOCK (1 word)- If the file is locked by the process inside a system call, this field specifies the PID of the process, otherwise, it is set to -1.
  • Unused (4 words)

All invalid entries are set to -1




Semaphore Table



The Semaphore Table contains details about all the semaphores used by the processes. It consists of MAX_SEM_COUNT (= 32 in the present eXpOS version) entries. Thus, there can be at most MAX_SEM_COUNT = 32 semaphores used in the system at any time. For every semaphore entry in the per-process resource table, there is a corresponding entry in the semaphore table.

A process can acquire a semaphore using the Semget system call, which returns a unique semid for the semaphore. Semid is basically the index of the semaphore in the semaphore table. Semget increments the Process count everytime it is invoked. The semaphore is initially not set, that is, Locking PID is -1. It locks it using SemLock system call. SemLock changes the Locking PID to the PID of the process invoking it. SemUnLock will reset (called release) the semaphore waking up all other processes which went to sleep trying to lock the semaphore. Finally when a process is done with the using of Semaphore, it releases it using Semrelease. Semrelease decrements the Process Count. A process may open several files and semaphores and execute the Fork system call multiple times to create an several process that shares all the semaphores acquired by the parent before the Fork system call.

Each entry is of size 4 words of which the last two are left unused. It follows the following format:

LOCKING PID PROCESS COUNT Unused
  • LOCKING PID (1 word)- specifies the PID of the process which has locked the semaphore
  • PROCESS COUNT (1 word) - specifies the number of processes waiting for semaphore
  • Unused (2 words)

All invalid entries are set to -1




Disk Status Table



Disk Status Table keeps track of the load and store operations. Every time a disk operation is invoked, the information regarding the disk transfer like the disk block and the memory page involved in the disk transfer, the process that invoked the transfer and type of disk operation are stored in Disk Status Table. The size of the table is 8 words of which the last 3 are unused.

Every entry of the Disk Status Table has the following format:

STATUS LOAD/STORE BIT PAGE NUMBER BLOCK NUMBER PID
  • STATUS (1 word) - specifies whether the disk is free or busy handling a memory-disk transfer.
  • LOAD/STORE BIT (1 word) - specifies whether the operation being done on the device is a load or store.
  • PAGE NUMBER (1 word) - specifies the page number of the page, which is being stored onto the disk or, to which a block from disk is being loaded.
  • BLOCK NUMBER (1 word) - specifies the block number of the disk block being loaded into memory or to which a page is being stored.
  • PID (1 word) - specifies the PID of the process who invoked the device. It has a value -1 if the disk transfer was initiated by the scheduler.



Buffer Table



To minimize the number of load and store operations, eXpOS provides a buffer cache in memory which would temporarily store disk blocks. Each disk block is mapped to a unique buffer. The disk blocks will be stored back to the disk when some other blocks replace it. Only modified blocks are written back to disk.

Buffer Table keeps the information about disk blocks present in the buffer. The table is updated whenever a block is loaded into buffer cache or when a buffer page is stored back into disk. There are MAX_BUFFER number of buffers available so the BUFFER TABLE will have that many entries. In this version of eXpOS, MAX_BUFFER = 10. Every entry is of 4 words of which 1 word is unused.

Every entry of the Buffer Table has the following format:

BLOCK NUMBER DIRTY BIT LOCKING PID Unused
  • BLOCK NUMBER (1 word) - specifies block number of the disk block stored in the buffer.
  • DIRTY BIT (1 word) - specifies whether the block stored in the buffer has been modified.
  • LOCKING PID (1 word) - specifies the PID of the process which has currently locked the buffer.
  • Unused (1 word)

Free entries are represented by -1 in all the fields.




System Status Table



It keeps the information about number of free pages in memory, number of processes waiting for memory pages and number of processes in swapped state. The size of the table is 4 words of which 1 word is unused.

The System Status Table has the following format:

MEM_FREE_COUNT WAIT_MEM_COUNT SWAPPED_COUNT Unused
  • MEM_FREE_COUNT (1 word) - specifies the number of free pages available in memory.
  • WAIT_MEM_COUNT (1 word) - specifies the number of processes waiting for memory.
  • SWAPPED_COUNT (1 word)- specifies the number of processes in SWAPPED state.
  • Unused (1 word)

Initially, when the table is set up by the OS startup code, the MEM_FREE_COUNT is initialized to **128, WAIT_MEM_COUNT to 0 and SWAPPED_COUNT to 0.




Terminal Status Table



Terminal Status Table keeps track of the Read operations done on the terminal. Every time a Read system call is invoked on the terminal, the information like the process that invoked the system call and status of the terminal (whether any process is reading from terminal or not) are stored in Terminal Status Table. The size of the table is 4 words of which the last 2 are unused.

Every entry of the Terminal Status Table has the following format:

STATUS PID Unused
  • STATUS (1 word) - specifies whether the terminal is free or is being used by a process to read input.
  • PID (1 word) - specifies the PID of the process who is currently reading input from the terminal.
  • Unused (2 words)



Memory Free List



The Memory free list is a data structure used for keeping track of used and unused pages in the memory. It consists of MAX_MEM_PAGE entries and each entry is of size one word. Thus, the total size of the free list is MAX_MEM_PAGE words. In this version of eXpOS, MAX_MEM_PAGES = 128

Each entry of the free list contains a value of either 0 indicating whether the corresponding page in the memory is free or number indicating the number of processes that share the page.
