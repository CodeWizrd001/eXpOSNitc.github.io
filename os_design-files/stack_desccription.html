Kernel Stack Management


This document is a tutorial explaining how the eXpOS kernel must manage the kernel stack. The document is specific to the implementation of eXpOS on the XSM machine. There are three different situations where the eXpOS kernel needs to do careful stack management:


   a. A user program invokes a system call

   b. An Exception / Interrupt occurs when a user process is executing

   c. A kernel module invokes another kernel module

System Call Invocation


An application program must push the input parameters to a system call through the user stack. The return value of the system call is also passed back to the application through the user stack.

If the application program is written in ExpL, then the eXpL compiler will generate code for pushing arguments into the user stack. The system call module must be designed so as to access these arguments from the stack of the process, do the processing required and store the return value into the appropriate place in the user stack before returning to the user mode. The system call module must also change the stack to its own (kernel) stack upon entry and switch back to the user stack at the time of return.


A user process before invoking a system call does the following actions :

   

Push the registers in use to the stack

   

Push the system call number into the stack

   

Push the arguments

  • Push one space for the return value

       

    Invoke the INT machine instruction corresponding to the system call

    1. The stack operations performed by the user program before INT instruction : The system call number corresponding to the system call is pushed onto the stack. Then the arguments are pushed in an order such that last argument comes on top. A push instruction PUSH R0 is done to put an empty space in the stack for the return value. The system call implementation must ensure that the return value is stored in this space. The interrupt routine needs this value to identify the system call. 




    
.... 			// Code to push registers
PUSH Function_Code	// Push system call number
PUSH Argument_1		// Push the arguments to the stack
PUSH Argument_2
PUSH Argument_3
PUSH R0			// Push an empty space for return value
INT number		// Invoke the corresponding INT instruction

    (NOTE: The machine code above is in the form of pseudo code.)


    2. After the INT instruction is executed : The INT instruction pushes the address of instruction after the INT instruction onto the stack. This value will be used by interrupt routine to return back from the system call to the next instruction in the calling program. The current value of SP register is stored in the USER STACK POINTER field of the user program's process table entry. The INT instruction changes mode from USER to KERNEL mode and passes control to the Interrupt Routine corresponding to the system call. The kernel then extracts the parameters from the application's stack and then changes the value of SP to the value in the KERNEL STACK POINTER. Then the instructions in the corresponding interrupt routine are executed.




    Before returning from a system call

    The return value is stored in the respective place in the user stack using the address in UPTR. The value of the SP register is then restored as the value in UPTR field.




    After return: The interrupt routine instruction transfers control back to the user program to the instruction following the INT instruction. The following machine instructions are present after the INT instruction in the eXpL compiled machine code given in the previous step.



    
POP R0			// Pop and save the return value
POP Argument_3		
POP Argument_2
POP Argument_1		// Pop and discard the arguments
POP Function_Code	// Pop and discard the system call number

    The machine code above pops the values from the stack. The system call number and arguments were inputs to the system call and hence they may be discarded now. The return value which is stored in the stack by the system call is fetched and used by the user program.





    • Interrupts or Exceptions


    A hardware interrupt/exception can occur while an application program is executing. It can be the exception handler, timer interrupt routine, disk interrupt routine and the console interrupt routine. Since the application does not have control over the transfer to the interrupt module, it would not have saved its context. Thus in this case the kernel module must save the register context of the application in its own stack (or elsewhere in the memory) before using the registers and must restore the context before returning to the application. The kernel stack is used to store the execution context of the user process. This context is restored before the return from the kernel module. (For the implementation of eXpOS on the XSM architecture, the backup and restore instructions facilitate this). There are four steps in handling exception/interrupt.

    1. Before the interrupt/exception occurs: The application process cannot store its context in the user stack as it doesn't have any control when an exception/interrupt occurs.



    2. After the interrupt/exception is executed: As soon as the control shifts to kernel, the current value of SP is stored in the User Stack Pointer entry of the process table and SP is set to the value in Kernel Stack Pointer. Once the execution mode is changed into kernel mode, it's the responibility of the kernel to backup the necessary data of the process. Thus, the context of the process is stored in the kernel stack. This changes the SP value (Note that the Kerenel Stack Pointer value remains the same resulting in an inconsistency). Now the corresponding exception/interrupt handler may be executed.



    3. Before the return: After the execution of the execution/interrupt handler, the context of the process is restored from the kernel stack. SP is set to the value of User Stack Pointer in the process table. The address of the next instruction to be executed is be obtained using the EIP register.



    4. After the return: The execution of the process continues with the next instruction after the one that resulted in the exception/interrupt.




    Kernel Module to Kernel Module Context Switch


    In this case, the current execution context is stored in the same stack as the working stack (kernel stack). This results in the value of SP to be incremented. This context is restored on returning back, after the execution of the next kernel module (For the implementation of eXpOS on the XSM architecture, the backup and restore instructions facilitate this). There are two steps in handling a kernel module to kernel module context switch.

    1. Before switching execution to the next kernel module: The current context is saved in the kernel stack.
